# 朝堂 · 流转协议与运行规则

## 一、阶段转换规则

朝堂的七个阶段不是线性必经之路。根据任务性质，可以跳过某些阶段。但阶段之间的转换有明确的触发条件和交接物。

### 转换矩阵

| 从 | 到 | 触发条件 | 交接物 |
|----|-----|---------|--------|
| Phase 1 分流 | Phase 2 议事 | 判定为复杂/模糊 或 用户说「议」 | 议题描述 + 背景信息 |
| Phase 1 分流 | Phase 4 规划 | 判定为明确 或 用户说「干」 | 目标描述 |
| Phase 2 议事 | Phase 3 决议 | 用户说「拍板」或 讨论自然收敛 | 三轮讨论记录 |
| Phase 3 决议 | Phase 4 规划 | 用户确认决议 或 5秒无异议 | 决议文档（共识+行动项） |
| Phase 4 规划 | Phase 5 审核 | 中书省完成方案 | 结构化执行方案 |
| Phase 5 审核 | Phase 4 规划 | 门下省 REJECTED | 审核意见 + 修改要求 |
| Phase 5 审核 | Phase 6 执行 | 门下省 APPROVED 或 用户说「过」 | 审核通过的方案 |
| Phase 6 执行 | Phase 7 交付 | 所有部门完成 或 用户说「交差」 | 各部门产出物 |

### 快速通道

当满足以下**全部**条件时，可跳过门下省审核（Phase 5），从中书省方案直接进入执行：
1. 任务只涉及单个部门
2. 无跨部门依赖
3. 用户明确表示不需要审核（说「干」或「过」）

快速通道下，中书省的方案输出可以简化——不需要完整的表格格式，用简洁的任务描述即可。

## 二、封驳循环管理

封驳是门下省的核心权力，但需要有边界：

### 循环计数
- 每次门下省 REJECT 计为一次封驳
- 计数器在方案通过或用户介入后重置

### 封驳上限处理（第 3 次被驳回）

总协调人将以下内容打包呈给用户：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ 方案已被封驳 3 次
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
中书省最新方案：
{方案摘要}

门下省历次审核意见：
第1次：{要点}
第2次：{要点}
第3次：{要点}

反复驳回的核心矛盾：
{总协调人分析}

请决定：
▸ 「干」— 以当前方案强制执行
▸ 「改」— 你来修改方案，我执行你的版本
▸ 「退」— 重新定义目标
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 三、Agent 子任务管理

### 生命周期

```
创建 → 发送初始消息 → 等待回复 → 收集结果 → 关闭
```

- 圆桌阶段的三位角色在整个讨论期间保持活跃，散会后关闭
- 三省的 Agent（中书/门下）是一次性的——完成即关闭
- 六部的 Agent 完成各自任务后即关闭

### 并行调度原则

| 场景 | 并行策略 |
|------|---------|
| 圆桌第一轮 | 三位角色同时启动 |
| 交叉质询 | 三位角色同时收到他人观点 |
| 中书+门下 | 串行——中书先出方案，门下再审 |
| 六部执行 | 无依赖的部门并行，有依赖的串行 |
| 多个部门的结果聚合 | 等待所有部门完成后统一聚合 |

### 超时处理

如果某个 Agent 子任务长时间无响应：
- 30 秒后检查一次状态
- 仍未完成则继续等待，向用户报告进度
- 如果明显卡住，尝试重新发送任务

## 四、持久化记忆

### 写入位置

| 数据类型 | 文件路径 | 写入时机 |
|---------|---------|---------|
| 团队档案 | `roundtable/team-profile.md` | 首次对齐 / 用户说「更新档案」 |
| 行动追踪 | `roundtable/action-tracker.md` | Phase 3 决议 / Phase 7 交付 |
| 会议纪要 | `roundtable/minutes/{日期}-{关键词}.md` | Phase 3 决议 / 散会 |
| 执行记录 | `chaotang-sessions/{日期}-{关键词}.md` | Phase 7 交付 |

### 执行记录模板

```markdown
# 执行记录：{任务名称}
> {日期} | 入口：{从哪个Phase开始} | 议事：{是/否}

## 目标
{一句话}

## 议事摘要（如有）
{决议要点}

## 执行方案
{方案标题}
封驳次数：{N}

## 参与部门与产出
| 部门 | 任务 | 产出 | 状态 |
|------|------|------|------|

## 交付物
{清单及位置}

## 遗留事项
{如有}
```

### 行动追踪表更新规则

追加行动项时：
```markdown
| {序号} | {行动项} | {责任人} | {截止日} | {来源会议/任务} | 待办 |
```

完成行动项时：
- 将状态从「待办」改为「已完成」
- 在行动项后追加完成日期

## 五、与圆桌会议 Skill 的关系

朝堂 = 圆桌议事 + 三省六部执行，是圆桌会议的超集。

- 如果用户只想讨论不想执行 → 用原来的 roundtable skill
- 如果用户想从讨论到执行一条龙 → 用朝堂
- 两者共享数据：team-profile.md、action-tracker.md、minutes/、roles/
- 朝堂的圆桌阶段遵循 roundtable skill 的议事规则，但由朝堂总协调人统一调度

如果用户在朝堂会话中说「散会」且还没进入执行阶段，按圆桌的散会流程处理（保存纪要、更新行动追踪、清理 Agent）。
